# 各个模块作用及其范围

本文档描述 Host 侧各模块的职责边界和当前实现状态。标准消息、命令、
topic、配置字段以 `CONTRACTS.md` 为权威表；本文只做模块范围说明。

状态标记：

- `[x]` 已实现并接入当前主链路。
- `[~]` 部分实现，或实现方式仍是临时/静态方案。
- `[ ]` 未实现或仍是未来扩展。

## 2. Watchdog

Watchdog 只盯当前主 agent，不直接操作设备对象，也不直接越过
AgentManager 调 node。

- [x] 注册/保存当前主 agent：用户选择主 agent 后，Watchdog 绑定该 agent。

- [x] 周期性发送 `check`：通过 `cmd_request` 发给 AgentManager。

- [x] 维护状态：`DISCONNECTED / INITIALIZING / HEALTHY / ERROR`。

- [x] 根据 `check` 和 `init_device` 结果做状态转换。

- [x] `init_device_req` 携带 `request_id`，并只接受匹配的 `cmd_result`。

- [x] 发布 `watchdog_state` 给 UI/消息总线。

- [x] 支持 `estop`：触发后回到 `DISCONNECTED`。

  ERROR` 不执行自动 cleanup/release/reset；需要实验人员人工处理后，通过
  `check` 失败回到 `DISCONNECTED`。



![image-20260603205348613](/home/jhxie/.config/Typora/typora-user-images/image-20260603205348613.png)

状态机：

```text
DISCONNECTED
  check 失败 -> DISCONNECTED
  check 成功 -> INITIALIZING

INITIALIZING
  init_device 成功 -> HEALTHY
  init_device 失败/超时 -> ERROR

HEALTHY
  check 成功 -> HEALTHY
  check 失败/estop -> DISCONNECTED

ERROR
  check 成功 -> ERROR
  check 失败 -> DISCONNECTED
```

## 3. AgentManager

AgentManager 是 Host 内部 agent 命令路由和 agent proxy 生命周期的拥有者。

- [x] 读取 `agents_config.json`，根据配置创建/复用 `AgentProxy`，区分 `primary_agents` 和其他 agents。
- [x] 初始化/释放 agent proxy。
- [x] 维护 agent proxy 集合，并记录当前 `active_agent_`。
- [x] 当前命令执行限制在 active agent 上；如果请求目标和 active agent 不一致，会返回失败。
- [x] 处理 `activate_agent / shutdown_agent / cmd_request / init_device_req / estop`。
- [x] 收到UI/watchdog 命令后，转发给目标 agent。 
- [x] 收集 agent 命令结果并发布 相关消息。
- [x] shutdown/release 时，AgentManager 会释放 AgentProxy；如果该 AgentProxy 曾启动本地 node 进程，则会结束本地 node。远程的则断开。

### 3.1 与其他模块的关系

- ScriptsActuator / master_CLI：通过 AgentManager 调用 agent 命令；当前脚本仍可直接连
  Python ActionServer，master_CLI 尚未落地。
- DataReceiver：不归 AgentManager 管。DataReceiver 负责 topic 订阅、频率统计和 UI 数据更新。
- AgentProxy：被 AgentManager 持有，是具体 node 的 Host 侧代理。
- Node：被 AgentProxy 启动或连接，执行真正设备逻辑。
- cmd_request      -> payload cmd/params        -> AgentProxy.cmd(cmd, params)
  init_device_req  -> config init_device_params -> AgentProxy.cmd("init_device", params)
  estop            -> {}                        -> AgentProxy.cmd("estop", {})

## 5. HostMessageBus

HostMessageBus 是 Host 内部线程安全消息队列。

- [x] 跨线程投递消息：
  
  UI -> AgentManager: activate_agent / cmd_request / shutdown_agent
  
  Watchdog -> AgentManager: cmd_request(check) / init_device_req / estop
  
  AgentManager -> Watchdog: cmd_result
  
  AgentManager -> UI: cmd_result / agent_activated / log_entry
  
  Watchdog -> UI: watchdog_state
  
  DataReceiver -> UI: topic_data
  
  ScriptsActuator -> UI: script_started / script_output / script_finished
  
  UI -> ScriptsActuator: run_script / stop_script
  
- [x] 支持目标队列：每个 target/consumer 有自己的队列。

- [x] 消息类型有权威表，见 `CONTRACTS.md`。

  [~] 支持订阅：当前是按 target 队列消费，不是按 topic/filter 的发布订阅系统。

- [x] 线程安全

- [ ] 通用 request/response 尚未实现，例如脚本请求 dialog 并等待用户返回。

## 6. ScriptsActuator / master_CLI

### 6.1 ScriptsActuator

ScriptsActuator 是 Host/master 级唯一脚本进程执行管理模块。

- [x] 通过线程安全消息总线接收 `run_script / stop_script`。
- [x] 根据 Python 脚本创建脚本进程。
- [x] 为脚本进程设置 `PYTHONPATH`、`RECORDLAB_AGENT`、
  `RECORDLAB_AGENTS_CONFIG`、`PYTHONUNBUFFERED` 等环境变量。
- [x] 收集 stdout/stderr。
- [x] 发布 `script_started`。
- [x] 发布带 `script_id / stream / process / script_path / pid` 的 `script_output`。
- [x] 发布带 `script_id / script_path / pid / exit_code` 的 `script_finished`。
- [x] 发布 `log_entry` 供 UI 日志展示。

边界：

- ScriptsActuator 拥有脚本进程生命周期。
- ScriptsActuator 不拥有 agent/node/device。
- ScriptsActuator 不直接处理 topic 数据。

### 6.2 master_CLI

master_CLI 的目标是接收脚本进程调用，并转为对 P0 master 的请求。
当前尚未落地；目前脚本仍可能直接连接 Python ActionServer。

- [ ] 发送标准 `cmd_request` / `agent_cmd`。
- [ ] dialog/input 用户输入。
- [ ] create_directory 目录创建。
- [ ] 把脚本侧能力统一收束到 Host/master。

## 9. Agents / AgentProxy

Agents 由 `agents_config.json` 配置，配置字段以 `CONTRACTS.md` 为准。

当前配置字段包括：

- `agents`
- `primary_agents`
- `name`
- `node_class`
- `process_type`
- `subnode_host`
- `action_name`
- `goal_port`
- `feedback_port`
- `data_port`
- `root_path`
- `topics`
- `custom_params`
- `init_device_params`
- `init_device_pause_duration`
- `default_scripts`

agent proxy当前字段中不包括：

- `subnode_path`
- `supported_devices`

AgentProxy 的职责是把 Host 内部统一的命令请求转换成对具体 node 的通信、
启动、连接、状态查询和结果回传。

### 

持有

AgentConfig config_;
std::string agents_config_path_; 
std::string nodes_root_;
std::string echo_python_root_;

std::unique_ptr<ProcessHandle> node_process_;
std::unique_ptr<EchoActionClient> action_client_;
bool connected_ = false;

- [x] 如果 node 是本地进程，AgentProxy 负责：

```text
launch node
检查 action server 是否可连接
terminate node
断开 action client
```

如果 node 是远程或外部进程，AgentProxy 不启动它，而是：

```text
connect to remote node
检测连接是否可用
断开 client
```
