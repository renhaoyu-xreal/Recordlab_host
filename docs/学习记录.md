## 2. `NodeRuntime` 运行逻辑

`NodeRuntime` 是通用节点运行时容器，负责加载、启动、运行、停止 agent 节点。

### 生命周期

```
main() → load_agent_config() → NodeRuntime(config)
  → signal 绑定 → runtime.start() → runtime.spin() → runtime.stop()
```

### 初始化 `__init__`

- 通过 `import_node_class` 动态加载节点类
- 检测 `requires_qt_event_loop` 决定是否创建 Qt 事件循环
- 实例化节点 + 创建 PublisherManager
- `node.bind_runtime(self)` 绑定回调

### 启动 `start()`

- `PublisherManager.start()` → 数据发布管道
- `ActionServer.start()` → 命令接收服务（监听 goal_port）

### 运行 `spin()`

- Qt 模式：`QTimer + app.exec()` 阻塞
- 无 Qt 模式：`while not _stopping: time.sleep(0.1)`

### 命令处理 `_on_goal()`

- 通过 `getattr(self.node, cmd)` 反射调用节点方法
- 统一包装返回 `{"success": bool, ...}`

### 停止 `stop()`

- `node.shutdown()` → `ActionServer.close()` → `PublisherManager.close()` → `qt_app.quit()`

**核心定位**：薄壳容器，不关心节点业务逻辑，只提供三个通用能力：
- 数据发布（PublisherManager）
- 命令接收与分发（ActionServer + 反射）
- 生命周期管理

## 3. Qt 事件循环 vs 无 Qt 方案

| 特性 | 无 Qt（默认） | Qt 方案 |
|------|-------------|---------|
| 代码 | `while not _stopping: time.sleep(0.1)` | `self.qt_app.exec()` |
| 适用 | 绝大多数节点 | 仅 BSP/MCU（XREAL SDK） |
| 优点 | 零依赖、极轻量、启动快 | 支持 QObject 信号槽 |
| 代价 | 无 | 引入 PySide6 运行时，内存开销更大 |

**BSP/MCU 需要 Qt 的原因**：XREAL SDK 是 Python Qt/QObject 实现，需要 Qt event loop 驱动信号槽。

`McuMainNode` 继承 `BspMainNode`，因此也继承 `requires_qt_event_loop = True`，MCU 同样走 Qt 方案。

## 4. `node_runtime.py` 启动时机

由 C++ Host 进程**按需启动为独立子进程**：

```
C++ Host → ACTIVATE_AGENT → ensureAgentConnected()
  → AgentProxy::launchOrConnect()
  → startNodeProcess()
  → fork + exec: python -m recordlab_nodes.core.node_runtime
      --config <配置> --agent <agent名>
```

### process_type 分类

| process_type | 运行方式 | 例子 |
|-------------|---------|------|
| `python_node` | Host fork 子进程 → `node_runtime.py` | bsp, mcu, imu_sim, nviz |
| `external_action` | 远程 TCP 连接 | UR_node |
| `local_scripts` | 本地脚本目录 | localhost |

**判断逻辑**（`agent_config_loader.cpp`）：
- 有 `node_class` → `python_node`
- 无 `node_class` 且无 `scripts_dir` → `external_action`
- 无 `node_class` 且有 `scripts_dir` → `local_scripts`

**每个 agent 一个独立子进程**，通过 ZeroMQ 端口通信，互不干扰。

## 7. 时间戳系统

### 两套时钟

| 时钟 | 起点 | 用途 |
|------|------|------|
| `time.time_ns()` | Unix 纪元 (1970-01-01) | 系统时间、录制时长 |
| `data.hmd_time_ns` | 设备上电/传感器初始化 | 传感器数据时标 |

### 偏移计算

```python
# 第一条 IMU 数据到达时
offset = time.time_ns() - hmd_time_ns  # 差值包含传输延迟
```

### 各时间戳来源

| 时间戳 | 来源 | 用途 |
|--------|------|------|
| `hmd_time_ns` | XR 眼镜 SDK | 传感器数据时标 |
| `sensor_time_offset_ns` | `now_ns - hmd_time_ns` | 硬件→系统时钟对齐 |
| `aligned_sensor_time_ns` | `hmd_time_ns + offset` | 对齐后的时间 |
| `delay_ns` | `now_ns - aligned` | 传输延迟监控 |
| `record_start_ns` | `time.time_ns()` | 录制时长计算 |
| `exposure_middle_time_ns` | SDK 相机参数计算 | 图像帧时标 |

### 偏移误差

| 连接方式 | 典型延迟 |
|----------|---------|
| USB 直连 | 1-5 ms |
| WiFi | 5-30 ms |
| Qt 信号排队抖动 | ±0.5 ms |

偏移的绝对误差在毫秒级，不是纳秒级精度。但数据时标用硬件时间戳，偏移只用于估算传输延迟，不参与数据打时标。

## 8. 录制与机械臂轨迹的时间对齐 & 算法评估

### 执行顺序（来自 nviz_ur_base.py）

脚本代码层面的调用顺序：

```
1. UR 移动到起始位        → 阻塞等待
2. 启动 UR 轨迹执行        → 非阻塞（远程 TCP 命令）
3. 立即启动眼镜录制        → 非阻塞（本机子进程命令）
4. 等待轨迹执行完成
5. 等待 2×delay 缓冲
6. 停止录制
7. 拷贝 UR 数据文件
```

### ⚠️ 实际启动时序的不确定性

代码调用顺序是 2→3，但**实际设备启动顺序不确定**：

- UR 命令通过 TCP 发往 `192.168.10.213`（远程机械臂控制器），网络延迟数毫秒到数十毫秒
- 眼镜录制命令通过 Host Bridge 发往本机 `node_runtime` 子进程，几乎是本机 IPC

**可能出现的情况**：

```
代码调用:    ──② UR轨迹 ──③ 眼镜录制──→
                
实际到达:    ────③ 眼镜录制(先到)─────→
                        ──② UR轨迹(后到)──→
```

眼镜录制命令大概率先于 UR 轨迹命令到达并开始执行。

### 录制 stop 时机的实际实现

```python
# nviz_ur_base.py 实际代码
delay = time_delay_func()              # 获取当前传输延迟 (ms)
                                       # 来源：TOPIC_TIME_DELAY → time_delay_ns / 1e6
delay_seconds = delay / 1000.0
wait_time = min(int(delay_seconds * 2 + 0.999), int(delay_seconds) + 10)
#            = min(ceil(2×delay_s), delay_s + 10)
#            典型值（delay=5ms）：min(1, 10) = 1 秒
#            极端值（delay=30s）：min(60, 40) = 40 秒

# 等待录制时长实际增长了 wait_time 秒
while wait_time > 0:
    record_time = record_time_func()        # 实时录制时长 (duration_ns / 1e9)
    if record_time >= last_record_time + wait_time:
        break                               # 录制时长真的增加了 wait_time
    time.sleep(1)
```

**关键机制**：不是盲 sleep，而是通过轮询 `TOPIC_RECORD_TIMER` 的 `duration_ns`，确认录制时长**确实增长了 wait_time 秒**才停止。即使停止命令有延迟，录制的额外缓冲时间也是实打实的。

### 为什么这不影响评估

但这**不影响算法评估**，因为：

1. **UR 轨迹文件自带独立时戳**：UR 控制器在轨迹执行过程中以自身时钟记录每帧 pose + timestamp，与命令何时到达无关
2. **眼镜数据自带独立时戳**：`hmd_time_ns` 从眼镜硬件时钟产生
3. **后处理只关心数据覆盖范围**：只要录制的时间窗口覆盖了 UR 轨迹数据的时间范围，就能在重叠段做对齐

```
UR 轨迹数据时戳范围:  ├──────────────────────┤
                         ↑ 可能早于录制开始
眼镜录制数据范围:           ├──────────────────────────┤
                            ↑ 本机响应快               ↑ 轮询 duration_ns 确保缓冲
                           
有效重叠段:                  ├──────────────────┤
                           (裁剪掉开头未覆盖的部分)
```

### 后处理对齐流程

```
UR 轨迹 (Ground Truth)  +  SLAM 估计轨迹
        ↓
1. 时间对齐 → 最近邻时间戳配对
2. 空间对齐 → Umeyama 刚性变换 (R, t, s)
3. 裁剪重叠窗口
4. 计算 ATE / RPE
```

# 时间对齐：最近邻时间戳配对 完整解释

## 1. 背景问题

你有两条独立轨迹：

- GT（UR 机械臂）：每一帧带时间戳 $t_g$

- SLAM 估计轨迹：每一帧带时间戳 $t_s$

两个设备**时钟独立**：

1. 开机时间不同，时间戳零点不一样；

2. 存在固定时间偏移（比如 SLAM 比 UR 晚启动 2.3 秒）；

3. 采样频率不同：UR 可能 100Hz，SLAM 30Hz，帧数不一一对应；

4. 没有统一触发同步信号，无法天然帧对帧匹配。

直接对比位姿毫无意义，必须先做**时间对齐**：找到物理上**同一时刻**发生的两组位姿，一一配对。

## 2. 核心思路：最近邻匹配

对 SLAM 里**每一个时间戳 $t_s$**，去全部 GT 时间戳列表里，找满足：

$$\min_{t_g \in GT\_times} |t_s - t_g|$$

也就是和当前 SLAM 帧时间差绝对值最小的那一帧 GT，作为配对。

### 通俗例子

- SLAM 帧时间：12.460s

- GT 候选时间：12.445 / 12.458 / 12.472
差值分别：0.015 / 0.002 / 0.012
最小差值是 12.458，这一 GT 帧和该 SLAM 帧配对。

## 3. 关键特性（对应你前面的疑问：和时钟原点无关）

### （1）不关心时间戳零点，只关心差值

不管：

- UR 时间戳从 10000 开始，SLAM 从 0 开始；

- 还是两者有固定全局偏移 $\Delta t$；

公式只算**两数之差**，全局偏移会被绝对值抵消，不影响谁是最近邻。

例：全部 GT 时间统一 + 2.5s，每一组 $|t_s - t_g|$ 差值不变，配对结果完全一样。

### （2）自动补偿固定时延

如果 SLAM 数据整体比 UR 晚 0.18s，所有 $t_s$ 天然比真实物理时刻慢 0.18s，最近邻匹配会自动匹配到滞后 0.18s 的 GT 帧，等价于自动修正固定时间偏移。

### （3）适配不同采样频率

GT 密、SLAM 稀疏：一个 GT 帧可能被多个 SLAM 帧匹配；

GT 稀疏、SLAM 密：多个 SLAM 帧匹配同一个 GT 帧；

算法不要求两边帧数相等。

## 4. 工程约束：时间阈值过滤（实际代码必加）

纯最近邻有缺陷：如果某一 SLAM 帧在 GT 前后很大一段区间都没有数据，会匹配一个相隔很远的帧，引入巨大误差。

因此实现时会加一步：

1. 找到差值最小的 $t_g$；

2. 判断 $|t_s - t_g| < max\_dt$（比如阈值 0.05s、0.1s）；

3. 超过阈值则丢弃该 SLAM 帧，不参与后续空间对齐与 ATE 计算。

作用：只保留时间上足够接近的有效配对，剔除无重叠区间的孤立帧。

### 空间对齐：Umeyama

时间对齐后得到 N 对匹配的 pose，用 Umeyama 算法求最优刚性变换：

$$ T^* = \arg\min_{R,t,s} \sum_{i=1}^{N} \| p_{gt}^i - (s \cdot R \cdot p_{est}^i + t) \|^2 $$

自动求解旋转 $R$（3×3）、平移 $t$（3×1）、缩放 $s$（标量，通常 $s \approx 1$）。

## 11. 自描述共享内存布局（Camera SHM v1）

> 记录日期：2026-06-23

### 背景

ROS1 风格分离架构：C++ host 进程 + Python node 进程，通过 **POSIX 共享内存** 传递摄像头预览图像。

Python 侧 `bsp_writers.py` 中的 `CameraSharedMemoryWriter` 是**定义方**，负责创建共享内存并写入版本化 header。
C++ 侧 `camera_shared_memory.cpp` 中的 `CameraSharedMemoryReader` 是**消费方**，读取 header 后自适应布局参数。

### 为什么需要自描述

共享内存两边是独立进程，没有 RPC 协商通道。如果 Python 升级了布局参数（slot 大小、编码方式等），C++ 若按编译期常量读取就会错位。解法是 Python 在共享内存开头写入一个 header，C++ attach 后先读 header 再决定如何解读后续数据。

### 内存布局

```
┌──────────────┬──────────────┬──────────────────────────────────────────┐
│ Header 64B   │ Seq 区       │ Slot 数据区                               │
│ (versioned)  │ cam×slot×8B  │ cam_count × slot_count × slot_size Byte  │
└──────────────┴──────────────┴──────────────────────────────────────────┘
offset 0       64             64 + cam_count × slot_count × 8
```

#### Header 结构 (64 bytes, little-endian uint32)

| 偏移 | 字段 | 说明 |
|------|------|------|
| 0 | magic | `0x52434D48` ("RCMH") |
| 4 | version | 布局版本号，当前 = 1 |
| 8 | cam_count | 摄像头数量 |
| 12 | slot_count | 每路环形缓冲 slot 数 |
| 16 | slot_size | 每个 slot 字节数 |
| 20 | meta_size | slot 头部元数据字节数 |

#### Slot 结构

每个 slot = meta_size(64B) + payload

| 偏移 | 字段 | 类型 | 说明 |
|------|------|------|------|
| 0 | width | uint32 | 图像宽度 |
| 4 | height | uint32 | 图像高度 |
| 8 | format | uint32 | QImage::Format 枚举 |
| 12 | data_size | uint32 | payload 实际字节数 |
| 16 | bytes_per_line | uint32 | 图像行步长 |
| 20 | encoding | uint32 | 0=raw 原始像素, 1=JPEG |
| 24-63 | reserved | - | 保留 |
| 64+ | payload | bytes | 图像数据 |

#### Seq 结构

每个 slot 对应一个 uint64 序列号，**数据写入完成后再更新 seq**。读取方先扫描所有 seq 找最新帧，读取 slot 数据后**二次校验 seq**（防止读到一半被覆盖的竞态）。

### 版本兼容流程

```
C++ attach()
  ├─ 读前 4 字节
  ├─ == kHeaderMagic (0x52434D48) → 版本化路径
  │   ├─ 从 header 读取 cam_count, slot_count, slot_size, meta_size
  │   └─ 按读取到的参数计算 seq/slot 偏移
  └─ ≠ kHeaderMagic → fallback 路径
      └─ 使用编译期硬编码默认值（kSlotSize 等），仅用于兼容旧版 Python
```

### 关键常量对照

| 常量 | Python (`bsp_writers.py`) | C++ (`camera_shared_memory.cpp`) | 同步方式 |
|------|--------------------------|----------------------------------|---------|
| Magic | `CAMERA_SHM_HEADER_MAGIC = 0x52434D48` | `kHeaderMagic = 0x52434d48` | 两侧写死相同值 |
| Slot Size | `CAMERA_SHM_SLOT_SIZE = 12*1024*1024` | 从 header 读；fallback `kSlotSize = 12*1024*1024` | Python 定义，C++ 自适应 |
| Meta Size | `CAMERA_SHM_META_SIZE = 64` | `kMetaSize = 64` | 两侧写死 64 |
| Slot Count | `CAMERA_SHM_SLOT_COUNT = 4` | 从 header 读 | Python 定义，C++ 自适应 |
| Cam Count | `CAMERA_SHM_CAM_COUNT = 2` | 从 header 读 | Python 定义，C++ 自适应 |
| Header Size | `CAMERA_SHM_HEADER_SIZE = 64` | `kHeaderSize = 64` | 两侧写死 64 |

### slot_size 选取依据

不同设备单帧原始数据量差异巨大：

| 设备 | 传感器 | 分辨率 | 原始数据量 | 编码 |
|------|--------|--------|-----------|------|
| GF | Rgb | 2016×1512 | ~9.1 MB | JPEG → ~300KB |
| Hylla | Slam | 640×512 | 320 KB | raw |
| Gina | Slam | 640×512 | 320 KB | raw |

slot_size 按**最大原始帧 + 余量**选取 = **12 MB**，确保 JPEG 压缩失败 fallback 到原始数据时也能装下。

### 实战教训

- 修改 Python 侧布局参数后，**必须重编 C++ host**（`./build.sh`），否则 fallback 默认值滞后会导致读偏移错位
- `QStringLiteral("共享内存图像数据不完整")` 等中文错误信息在 `strings` 命令中不可见（UTF-16 编码），不代表未编译进去
- 十六进制字面量 `0x4D` ≡ `0x4d`，大小写不敏感，不必统一
- Python 和 C++ 两边都有同样名字的常量不代表它们会"自动同步"——分离进程架构下，唯一的同步渠道就是这个 versioned header

## 12. 案例：SDK 返回伪 RGB 图像的排查

> 记录日期：2026-06-23
> 设备：GF（X1 芯片，HEVC 硬解码 RGB 摄像头）

### 现象

RGB 画面能显示，但图像异常：

1. **图片顶部有一条红绿色交替变化的横线**
2. **画面中有几块淡淡的红色、蓝色污渍**
3. 流水线所有数值检查（format、bpl、encoding）全部正常

### 排查过程

#### 第一步：排除共享内存布局问题

之前已定位并修复了 Python/C++ 共享内存布局不一致（slot_size 不匹配 → 数据读偏移错位）。重编后验证：

```
C++ 日志: camera shm render cam=0 size=2016x1512 bytes=166107 encoded=1
```

`encoded=1`（JPEG）、尺寸 2016×1512、字节数 ~160KB，全对。**共享内存传输层没有问题**。

#### 第二步：加调试日志，逐段排查图像流水线

在图像管线的三个关键点加了首帧日志：

| 检查点 | 位置 | 观察内容 |
|--------|------|---------|
| SDK 原始 QImage | `bsp_device._on_cam_data` | format、bpl、isNull |
| convertToFormat 后 | `bsp_device._on_cam_data` | 转换后的 format、bpl、isGray |
| 写入共享内存前 | `bsp_writers.write_qimage` | 输入/输出对比、编码方式、data_size |

日志结果：

```
[BspDevice] cam source image:  idx=0 width=2016 height=1512 format=13 bpl=6048 isNull=False
[BspDevice] cam converted image: idx=0 target=rgb888 width=2016 height=1512 format=13 bpl=6048 isGray=False bsp_stride=2016
[CameraSHM] write_qimage cam=0 input: fmt=13 isGray=False bpl=6048 size=2016x1512 | output: fmt=13 bpl=6048 size=2016x1512 encoded=JPEG data_size=166107
```

- `format=13` = `QImage::Format_RGB888` ✓
- `bpl=6048` = 2016×3 ✓
- `isGray=False` ✓
- `encoded=JPEG` ✓

**QImage 对象在语义层面完全正确**——格式声明、步长、尺寸、编码方式全都对。

#### 第三步：排除法缩小范围

既然 QImage 的元数据全部正确，问题一定在 **QImage 包裹的实际像素数据**上。QImage 有两种构造方式：

1. **拥有数据**：`QImage(width, height, format)` → Qt 自己分配 buffer，`convertToFormat` 会做真正的像素格式转换
2. **包装外部数据**：`QImage(ext_buffer, width, height, bpl, format)` → 不拥有数据，只是指向外部 buffer 的视图

XREAL SDK 的 `bsp_img.image` 很可能是方式 2——把 HEVC 解码器的输出 buffer 包装成 QImage。HEVC 输出的是 **YUV420**，但 SDK 声明的 format 却是 **RGB888**。

#### 第四步：从现象反推数据格式

QImage 语义是 RGB888 → `convertToFormat(RGB888)` 认为不需要转换 → 直接拷贝原始字节 → JPEG 编码器也按 RGB888 压缩 → C++ 按 JPEG 解码 → 显示。

但实际上原始字节是 YUV420，被当作 RGB888 解读：

```
YUV420 在内存中的布局（2016×1512）：
┌────────────────────────────────────┐
│ Y 平面: 2016×1512 bytes            │  ← 亮度，每像素 1 字节
│ (每个字节 = 一个像素的亮度)         │
├────────────────────────────────────┤
│ UV 平面: 2016×756 bytes            │  ← 色度，4:2:0 子采样
│ (U V 交错，每 4 个像素共享一组 UV)  │
└────────────────────────────────────┘
总大小 = 2016×1512 + 2016×756 = 2016×1512×1.5 ≈ 4.6 MB

RGB888 在内存中的布局（2016×1512）：
┌────────────────────────────────────┐
│ R G B R G B R G B ...              │  ← 每像素 3 字节
│ 共 2016×1512×3 = 9.1 MB            │
└────────────────────────────────────┘
```

**关键数值对比**：YUV420 = 4.6 MB，RGB888 = 9.1 MB。数据量差了一倍，而 QImage 的 `bytesPerLine()` 报告 6048（即 RGB888 的行步长）。这意味着 Qt 会按 6048 字节/行去读数据，但实际只有 YUV 的约 3024 字节/行有意义，超出部分是**未初始化内存或相邻 buffer 的垃圾数据**。

#### 第五步：用现象验证假设

| 现象 | YUV 被当作 RGB 的对应表现 |
|------|--------------------------|
| **顶部红绿交替线** | 第一行 Y（亮度）数据被当作 R 通道 → Y 值高的像素呈绿色。相邻行的 UV 数据混入 → 红色。两行交替 → 红绿色交替线 |
| **红蓝污渍** | UV 色度平面（子采样到 1/4 分辨率）落在画面某些区域 → 被当作 RGB 的 G/B 通道 → 局部偏红/偏蓝 |
| **整体画面可见** | Y 通道本身携带完整亮度信息 → 轮廓、明暗大致正确（因为亮度在 RGB 中也主导感知） |

### 根因

**XREAL SDK（GF 设备）的 `bsp_img.image` 返回的 QImage，声明 format=RGB888、bpl=6048，但底层 buffer 实际存放的是 HEVC 解码器输出的 YUV420 数据。** SDK 没有做 YUV→RGB 转换，直接把 YUV buffer 包了一层 RGB888 的 QImage 壳子。

### 解决方向

1. 检查 SDK 的 `bsp_img` 是否有原始 format 字段（如 `pixel_format`、`fourcc`），确认真实像素格式
2. 如果确认是 YUV420，在 Python 侧手动用 `cv2.cvtColor(yuv_data, cv2.COLOR_YUV2RGB_NV12)` 转换后再构造 QImage
3. 或者用 QImage 的 `reinterpretAsFormat()` 探测真实格式

### 方法论总结

```
现象（红绿线 + 污渍）
  → 加日志检查每层元数据（format, bpl, isGray）
  → 元数据全对 → 问题在数据本身
  → QImage 不拥有数据（包装 SDK buffer）
  → 对比 YUV420 vs RGB888 数据量（1.5× vs 3× bytes/pixel）
  → 现象与 YUV 被误读为 RGB 的特征完全吻合
  → 根因：SDK 声明了错误的 QImage format
```

核心经验：**QImage 的 format 字段是"声明"，不代表像素数据真的长那样。** 对于包装外部 buffer 的 QImage（不拥有数据的场景），format 是调用方填的，SDK 填错了就是错了。排查时要区分"QImage 对象语义"和"底层 buffer 实际内容"。

## 机械臂停止为什么不能只靠“高优先级字段”

### 现象

- UI 点击“停止执行”后，会立即显示“已请求机械臂急停”
- 但机械臂实际并没有马上停下
- 日志里能看到脚本停止流程的 UI 日志，却看不到 `AgentManager` 真的发出了 `cmd=estop`

### 根因

之前的 Host 架构里，所有命令都进同一个 `AgentManager` 队列，由同一个 worker 串行处理：

1. `execute_trajectory` 发出去后，`AgentManager` 会同步阻塞等待它返回
2. 此时 UI 再点“停止执行”，`estop` 虽然被 `publish` 到总线，但还是排在同一个消费者后面
3. 因为取队列的人就是那个已经被 `execute_trajectory` 卡住的 worker，所以 `estop` 根本抢不到执行机会

也就是说，**问题不是队列里有没有高优先级消息，而是根本没有空闲的执行通道去消费它。**

### 第一轮结论

只有“普通队列 + 高优先级队列”还不够，必须同时满足下面两点：

1. 高优先级消息要进入独立队列
2. 高优先级队列要有独立 worker，并且不能复用正在被长命令占住的那条阻塞 command 通道

否则“高优先级”只是名字高，实际还是要等前面的阻塞命令结束。

### 第二轮修正

后续联调又暴露出一个新问题：`check` 虽然不该被长命令堵住，但它本质上是**高频健康轮询**，并不属于“抢占式控制命令”。

如果把 `check` 也塞进最高优先级通道，会带来两个副作用：

1. `check` 频率高，会挤占真正紧急的 `estop`
2. 如果健康检查 client 的生命周期处理不好，还会放大 ZMQ 连接的创建/销毁抖动，表现成 watchdog 假性 `disconnect`、`地址已在使用`，甚至 `signaler.cpp` 断言

所以最终结构不是“两层”，而是“三层”：

- `agent_manager`
  普通命令通道
- `agent_manager_health`
  健康检查通道，专门给 `check`
- `agent_manager_priority`
  抢占式控制通道，专门给 `estop` / `stop_record` / `start_device` / `reboot_device`

### 最终实现

本次最终把 Host 侧命令链路整理成了真正的三通道结构：

- 新增总线目标：
  - `agent_manager_health`
  - `agent_manager_priority`
- `AgentManager` 从单 worker 改成：
  - 普通 worker：消费 `agent_manager`
  - 健康检查 worker：消费 `agent_manager_health`
  - 抢占控制 worker：消费 `agent_manager_priority`
- `check` 不再走高优先级，而是走独立 health worker
- `estop` 仍然保留在最高优先级控制通道
- `health` 与 `priority` 各自维护独立长生命周期 `EchoActionClient`
  避免频繁创建/销毁 ZMQ fixed-port client 导致崩溃

### 现在的通道分层

- 普通通道 `agent_manager`
  - 常规 UI 命令
  - 普通脚本命令
  - `init_device`
- 健康检查通道 `agent_manager_health`
  - Watchdog `check`
- 抢占控制通道 `agent_manager_priority`
  - `estop`
  - `stop_record`
  - `start_device`
  - `reboot_device`
  - 脚本 runtime 中显式声明 `priority=high` 的命令

### 对以后排查的提示

如果再次出现“UI 说已停止，但设备没停”的问题，先不要只看 UI 日志，而要分三层确认：

1. `ScriptsActuator/Watchdog` 是否把消息发到了正确的通道
2. 对应 worker 是否真的取到了这条消息
3. 对应 worker 的长生命周期 client 是否正常可用并真正发出了设备命令

这个问题本质上不是“有没有高优先级”这么简单，而是**阻塞执行链路、健康检查链路、抢占控制链路是否被正确隔离**。以后设计新命令时，先判断它属于哪一种，再决定该进哪条通道。

## 2026-06-26 停止脚本后 `signaler.cpp:265` 断言闪退

### 现象

停止 nviz 录制脚本后，UI 已经提示“脚本已停止并完成收尾”，但主程序几秒后仍然闪退，日志末尾类似：

- `ProcessHandle terminated process pid=...`
- `Assertion failed: pfd.revents & POLLIN (src/signaler.cpp:265)`

而且这次崩溃点已经不再是 watchdog 误触发 `init_device`，而是发生在 `RELEASE_INACTIVE_AGENTS` 清理临时 agent 的过程中。

### 根因

排查 `echo_message_system` 的 fixed-port `ActionClient` 后发现，析构顺序原来是：

1. `fixed_running_ = false`
2. 先 `close()` goal / feedback socket
3. 最后才 `join()` feedback 监听线程

但监听线程此时可能还阻塞在 `fixed_feedback_socket_->recv(...)`，也就是说 **socket 先被关闭，线程还没退出**。这会在 ZMQ 内部触发 `signaler.cpp:265` 断言，表现成 stop 后几秒闪退。

### 修复

把 fixed-port `ActionClient` 析构顺序改成：

1. `fixed_running_ = false`
2. 先 `join()` 监听线程
3. 再 `close()` / `reset()` sockets
4. 最后释放 context

对应文件：

- [third_party/echo_message_system/cpp-refactor/src/action.cpp](/home/hyren/Recordlab_host/third_party/echo_message_system/cpp-refactor/src/action.cpp:213)

### 排查经验

以后如果再遇到“停止脚本后隔几秒闪退”，优先区分两类问题：

1. 如果 `SCRIPT_FINISHED` 后马上出现 watchdog 的 `DISCONNECTED -> INITIALIZING -> init_device`，先查 UI/watchdog 绑定逻辑
2. 如果 `SCRIPT_FINISHED` 后是在 `ProcessHandle terminated process` 附近崩溃，优先查临时 agent 对应的 `ActionClient` / ZMQ 线程销毁顺序
