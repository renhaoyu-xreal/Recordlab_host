# 打包、发布、部署更新

这份文档把操作人员最常用的命令和注意事项放在最前面，后面再解释原理。

## 1. 命令速查

### 1.1 打包前先检查 `old_video.mp4`

`old_video.mp4` 路径：

```text
third_party/Recordlab_nodes/node_scripts/localhost/old_video.mp4
```

这个文件比较大，当前没有纳入 git 管理。

如果这次发布需要视频播放相关功能，打包前要手动把它放回上面的目录，再执行打包。

打包没有这个文件会报warning

### 1.2 开发机打包

```bash
cd Recordlab_host
./host_scripts/package_bin_release.sh
```

如果已经提前编译过，也可以：

```bash
./host_scripts/package_bin_release.sh --skip-build
```

### 1.3 通过网络发布到服务器

第一次同步有4.1G视频会很慢，建议首次同步u盘拷到服务器，后续更新只同步变化的部分，就会很快

```bash
cd Recordlab_host
chmod +x ./host_scripts/deploy_bin_release.sh
./host_scripts/deploy_bin_release.sh
```

如果已经打过包，不想重复打包：

```bash
./host_scripts/deploy_bin_release.sh --skip-package
```

### 1.4 实验室机器第一次拉取

第一次同步有4.1G视频会很慢，建议首次同步u盘拷到实验室机器，后续更新只同步变化的部分，就会很快

```bash
mkdir -p ~/proj/Recordlab_host
rsync -a nreal@10.2.11.200:/home/nreal/nviz_record_data/Recordlab_host/ ~/proj/Recordlab_host/
```

要求：

- 本机目录应当是 `~/proj/Recordlab_host`
- 目录内应直接看到 `RecordLabHost.sh`
- 不要再多包一层 `RecordLabHostBin`

### 1.5 实验室机器启动

```bash
cd ~/proj/Recordlab_host
bash ./host_scripts/install_dependencies.sh
./RecordLabHost.sh
```

说明：

- 实验室机器第一次启动，建议先手动执行一次 `host_scripts/install_dependencies.sh`
- 这个脚本会补系统依赖、创建 Python 环境，并完成首次构建

### 1.6 更新版本号和更新说明

编辑：

```text
third_party/Recordlab_nodes/config/recordlab_product_config.jsonc
```

主要改：

```jsonc
"version": "v2.0.1",
"update_info": "这里写本次更新说明"
```

### 1.7 调整设备启动逻辑

优先看这两个配置文件：

```text
third_party/Recordlab_nodes/config/agents_config.json
third_party/Recordlab_nodes/config/recordlab_product_config.jsonc
```

改完以后标准动作：

1. 本地确认配置没写坏
2. 重新打包
3. 重新发布
4. 用一台实验室机器先验证

## 2. 总体流程

当前流程分 3 段：

1. 开发机在 `Recordlab_host` 仓库里打包
2. 开发机把打包结果发布到服务器目录
3. 实验室机器第一次手动拉取；之后每次启动自动从服务器同步更新

服务器固定发布目录：

```text
/home/nreal/nviz_record_data/Recordlab_host
```

实验室机器推荐安装目录：

```text
~/proj/Recordlab_host
```

## 3. 如何打包

在开发机执行：

```bash
cd Recordlab_host
./host_scripts/package_bin_release.sh
```

如果已经提前编译好了，也可以复用当前 `build/`：

```bash
./host_scripts/package_bin_release.sh --skip-build
```

打包完成后会生成：

```text
dist/RecordLabHostBin
dist/RecordLabHostBin-linux-x86_64.tar.gz
```

### 3.1 打包脚本做了什么

打包入口是 `host_scripts/package_bin_release.sh`。

它会先检查 `rsync` 是否存在，然后：

1. 默认调用 `host_scripts/build.sh`
2. 清理旧的 `dist/RecordLabHostBin`
3. 复制运行和重建所需文件
4. 生成发布包专用启动脚本
5. 生成压缩包

### 3.2 具体打包了什么

当前发布包不是“纯二进制包”，而是“可运行 + 可重新构建”的完整运行目录，包含：

- `bin/recordlab_host_app`
- `bin/recordlab_cli`
- `bin/start_recordlab.sh`
- `bin/update_from_server.sh`
- `RecordLabHost.sh`
- `CMakeLists.txt`
- `app/`
- `include/`
- `src/`
- `config/`
- `third_party/Recordlab_nodes/`
- `third_party/echo_message_system/`
- `third_party/xreal_glasses/`，如果当前仓库下存在
- `host_scripts/build.sh`
- `host_scripts/install_dependencies.sh`
- `docs/用户手册/`
- `README.md`

注意：

- 不会打包整个 `docs/`，只会打包 `docs/用户手册/`
- 不会打包 `.git/`、`build/`、`dist/`、`logs/`、`data/`、缓存文件
- 不会打包 `third_party/Recordlab_nodes/` 根目录下的 `nrsensor_*.txt`
- `old_video.mp4` 只有在你手动放回 `third_party/Recordlab_nodes/node_scripts/localhost/` 时才会被打包
- 如果打包时缺少 `old_video.mp4`，脚本会输出 warning，但不会中断打包

### 3.3 `old_video.mp4` 相关说明

当前文件位置应为：

```text
third_party/Recordlab_nodes/node_scripts/localhost/old_video.mp4
```

说明：

- 这个文件体积较大，没有放进 git
- 打包脚本会整包复制 `third_party/Recordlab_nodes/`，所以只要它在目录里，就会被一起打进发布包
- 如果目录里没有它，发布包里也不会有它
- 打包脚本现在会对这种情况输出 warning，提醒操作人员先手动补文件

## 4. 如何发布到服务器

推荐使用一键发布脚本：

```bash
cd Recordlab_host
./host_scripts/deploy_bin_release.sh
```

如果已经打过包，不想重复打包：

```bash
./host_scripts/deploy_bin_release.sh --skip-package
```

### 4.1 发布脚本做了什么

发布入口是 `host_scripts/deploy_bin_release.sh`。

它会：

1. 默认先调用 `host_scripts/package_bin_release.sh`
2. 确保服务器目录存在
3. 在服务器目录下创建锁文件 `.publish_in_progress`
4. 用 `rsync` 把 `dist/RecordLabHostBin/` 同步到服务器
5. 结束时删除锁文件

实际同步命令在 `host_scripts/deploy_bin_release.sh` 中：

```bash
sshpass -p "${REMOTE_PASSWORD}" rsync -a --delete \
  --exclude ".publish_in_progress" \
  "${PACKAGE_DIR}/" "${REMOTE_HOST}:${REMOTE_ROOT}/"
```

当前默认发布目标：

- 主机：`nreal@10.2.11.200`
- 目录：`/home/nreal/nviz_record_data/Recordlab_host`
- 密码：`1`

可通过环境变量覆盖：

- `RECORDLAB_RELEASE_HOST`
- `RECORDLAB_RELEASE_ROOT`
- `RECORDLAB_RELEASE_PASSWORD`

### 4.2 为什么要有 `.publish_in_progress`

实验室机器启动时会自动从服务器拉更新。

如果服务器正在发布，实验室机器此时直接拉文件，可能会拿到一半旧文件、一半新文件。

所以发布脚本会先创建：

```text
/home/nreal/nviz_record_data/Recordlab_host/.publish_in_progress
```

实验室机器看到这个文件，就会跳过这次自动更新，继续使用本机已有版本。

## 5. 实验室机器第一次拉取应该怎么做

第一次部署时，实验室机器不会自己“凭空得到这个软件”，需要先手动从服务器复制一份到本机。

推荐步骤：

```bash
mkdir -p ~/proj/Recordlab_host
rsync -a nreal@10.2.11.200:/home/nreal/nviz_record_data/Recordlab_host/ ~/proj/Recordlab_host/
```

或者：

```bash
mkdir -p ~/proj
cp -a /path/from/temporary/copy ~/proj/Recordlab_host
```

关键要求：

- 本机最终目录应该是 `~/proj/Recordlab_host`
- 不要在 `~/proj/Recordlab_host` 里面再多包一层 `RecordLabHostBin`
- 目录内应该直接能看到 `RecordLabHost.sh`

完成后，实验室机器通过下面命令启动：

```bash
cd ~/proj/Recordlab_host
bash ./host_scripts/install_dependencies.sh
./RecordLabHost.sh
```

## 6. 第一次打开会发生什么

实验室机器第一次运行 `./RecordLabHost.sh` 时，会按下面顺序执行。

### 6.1 先尝试自动更新

发布包根入口 `RecordLabHost.sh` 会先执行 `bin/update_from_server.sh`，再执行 `bin/start_recordlab.sh`。

对应生成逻辑在 `host_scripts/package_bin_release.sh` 中。

`update_from_server.sh` 会：

1. 检查是否设置了 `RECORDLAB_SKIP_AUTO_UPDATE=1`
2. 检查本机是否能免密 SSH 到 `nreal@10.2.11.200`
3. 检查服务器目录是否存在
4. 检查服务器上是否有 `.publish_in_progress`
5. 用 `rsync -a --delete` 从服务器同步到本机

实际拉取命令在 `host_scripts/package_bin_release.sh` 中：

```bash
rsync -a --delete \
  --exclude ".venv-py310/" \
  --exclude "logs/" \
  --exclude "data/" \
  --exclude ".git/" \
  "${REMOTE_HOST}:${REMOTE_ROOT}/" "${APP_ROOT}/"
```

说明：

- 本机 `.venv-py310/`、`logs/`、`data/` 不会被服务器覆盖
- 如果自动更新失败，只会弹窗提示，不会阻止程序继续启动本机已有版本

### 6.2 先检查关键文件，再检查 Python 依赖

`bin/start_recordlab.sh` 会先检查：

- `third_party/Recordlab_nodes/recordlab_nodes/core/node_runtime.py`
- `third_party/echo_message_system/python/message_system`
- `third_party/Recordlab_nodes/config/agents_config.json`
- `recordlab_nodes`
- `message_system`

是否存在、是否能正常导入。

如果关键文件存在，但 Python 包不能导入，或者需要首次构建，就会自动执行：

```bash
./host_scripts/install_dependencies.sh
```

这一步通常发生在：

- 第一次启动
- 本机 `.venv-py310` 被删掉
- Python 环境损坏
- `recordlab_host_app` 不存在，或需要重新构建

这一步不只是“装 Python 包”，还会检查 Qt6 Widgets、`zmq.hpp` 等 C++ 构建依赖，并在最后执行一次完整构建。

### 6.3 再检查是否需要自动构建

如果出现以下任一情况，程序会自动调用：

```bash
./host_scripts/build.sh
```

触发条件：

- `bin/recordlab_host_app` 不存在
- `CMakeLists.txt`、`app/`、`include/`、`src/` 中有文件比 `bin/recordlab_host_app` 新

这是为了保证实验室机器同步到新源码后，必要时能在本地重建 host。

### 6.4 最后启动主程序

依赖和构建都准备好后，才会启动 UI。

## 7. 后面每次打开会发生什么

后续每次执行 `./RecordLabHost.sh`，仍然会走同一套流程，但通常会更快：

1. 先从服务器拉最新文件
2. 发现 `.venv-py310` 还在，跳过依赖安装
3. 发现 `bin/recordlab_host_app` 仍然是最新，跳过构建
4. 直接启动 UI

只有在下面这些情况下，后续启动才会再次变慢：

- 服务器上刚发布了新代码
- 本机 Python 环境损坏
- 本机二进制缺失
- 需要根据同步到的新源码重新 build

## 8. 更新信息怎么改

更新信息主要来自：

```text
third_party/Recordlab_nodes/config/recordlab_product_config.jsonc
```

这个文件当前由 `config/host_runtime.json` 里的 `product_config_path` 指向：

```json
"product_config_path": "third_party/Recordlab_nodes/config/recordlab_product_config.jsonc"
```

### 8.1 要改哪两个字段

在 `third_party/Recordlab_nodes/config/recordlab_product_config.jsonc` 里主要改：

```jsonc
"version": "v2.0.1",
"update_info": "1、优化眼镜连接速度、稳定性、异常处理\n2、删除启动按钮，全部自动连接、重连\n3、增加结果目录显示、导出功能"
```

含义：

- `version`：显示在主界面状态栏
- `update_info`：程序启动后弹出的“更新信息”内容

### 8.2 修改后的生效方式

流程是：

1. 修改 `recordlab_product_config.jsonc`
2. 重新打包
3. 重新发布到服务器
4. 实验室机器下次启动自动同步后生效

## 9. 如何通过两个配置文件调整设备启动逻辑

当前建议重点看这两个文件：

1. `third_party/Recordlab_nodes/config/agents_config.json`
2. `third_party/Recordlab_nodes/config/recordlab_product_config.jsonc`

这两个文件分工不同。

### 9.1 `agents_config.json` 负责每个 Agent 的启动入口和默认行为

路径：

```text
third_party/Recordlab_nodes/config/agents_config.json
```

常见可调整字段：

- `init_device_params`
- `watchdog_start_device`
- `default_scripts`
- `custom_params`
- `topics`
- `sensor_layout`

#### 例 1：调整 NViz 初始化参数

在 `glasses_nviz_node` 里当前是：

```json
"init_device_params": {
  "enable_udp": false,
  "enable_tcp": true
}
```

如果你要调整 NViz 初始化时默认启用的通信方式，就改这里。

#### 例 2：调整 Android 初始化端口

在 `android` 里当前是：

```json
"init_device_params": {
  "tcp_port": 8100
}
```

如果 Android 端口改了，就改这里。

#### 例 3：调整 Nebula 启动逻辑

在 `nebula_trial` 里当前是：

```json
"init_device_params": {
  "enable_wifi_adb": true,
  "require_ping": true,
  "setenforce_permissive": true,
  "adb_port": 5555,
  "remote_dir": "/sdcard/3dof_data"
}
```

如果要改 Nebula 初始化流程，比如是否强制 ping、ADB 端口、远端录制目录，就改这里。

#### 例 4：调整是否由 Watchdog 自动拉起设备

很多 agent 当前都设置了：

```json
"watchdog_start_device": false
```

如果将来要改成切换 agent 后由 Watchdog 更主动地参与拉起流程，就需要看这个字段。

### 9.2 `recordlab_product_config.jsonc` 负责设备型号和默认启动参数

这个文件除了 `version` / `update_info`，还维护了：

- `usb_product_catalog`

其中一些设备条目下有：

```jsonc
"bsp": {
  "default_start_device_params": {
    "camera_mode": "rgb",
    "rgb_grayscale_mode": true,
    "rgb_fps": 10,
    "rgb_auto_exposure": false,
    "rgb_exposure": 400
  }
}
```

这个配置的作用是：

- 针对不同眼镜型号
- 给 BSP / NViz 相关设备提供默认启动参数
- 影响设备启动时默认使用的相机模式、帧率、曝光等

适合改这里的内容包括：

- 某型号默认 `camera_mode`
- 默认 `rgb_fps`
- 默认曝光参数
- 默认连接方式 `default_connection`

### 9.3 修改配置后的标准动作

只要修改了上述两个配置文件之一，都建议按下面顺序做：

1. 本地确认 JSON/JSONC 没写坏
2. 重新打包
3. 重新发布到服务器
4. 在一台实验室机器上先验证
5. 再通知其他机器正常使用

## 10. 服务器操作人员的日常建议

- 发布前先确认当前仓库内容正确
- 重要变更先改 `version` 和 `update_info`
- 发布尽量避开实验室机器正在大量启动的时间段
- 发布后最好在一台实验室机器上手动启动验证一次
- 不要手动删除服务器目录中的 `.publish_in_progress`，除非确认发布脚本异常退出

## 11. 最短操作清单

### 11.1 只改更新提示

1. 修改 `third_party/Recordlab_nodes/config/recordlab_product_config.jsonc`
2. 执行 `./host_scripts/deploy_bin_release.sh`
3. 让实验室机器重启程序

### 11.2 改设备启动逻辑

1. 修改 `agents_config.json` 或 `recordlab_product_config.jsonc`
2. 本地测试
3. 执行 `./host_scripts/deploy_bin_release.sh`
4. 用一台实验室机器验证

### 11.3 实验室新机器首次部署

1. 从服务器复制 `Recordlab_host` 到 `~/proj/Recordlab_host`
2. 执行 `./RecordLabHost.sh`
3. 等待自动更新、依赖安装、自动构建完成
4. 进入 UI 后做一次基本验证
